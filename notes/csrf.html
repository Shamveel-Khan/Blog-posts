What is CSRF?

Preconditions (what must be true)

  • The site accepts the request and makes a change without a proper check.

Common attack tricks

Quick Burp test


Capture a real change-email request in Proxy.

Forge the same request from an exploit page (auto-submit form or redirect).

If it changes the account, the site is vulnerable.


Quick swap test

  • Use two accounts. Swap the cookie or CSRF value between them in Repeater.

  • If swapping lets you change the other account, the token isn‚Äôt tied to the session.

Why protections fail (short)

  • No server-side token check ‚Äî site just uses cookies.

  • Token is compared to a cookie the attacker can set.

  • Site accepts GET or uses method-override for state changes.

  • Some endpoint allows Set-Cookie injection (attacker can set victim cookie).

What stops CSRF (practical fixes)


Server-side CSRF token per user session, verified on every unsafe request. (Best fix.)

Do not let GET requests change data. Use POST/PUT for changes.

Sanitize inputs ‚Äî do not reflect user input into response headers.

Use SameSite on cookies as extra protection, but not the only protection.
